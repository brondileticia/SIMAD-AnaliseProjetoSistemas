Do Problema ao Escopo

A Defesa Civil de Campo Limpo Paulista operava com fragmentação operacional: dados climáticos, vistorias técnicas e alertas residiam em canais independentes e desconectados.

Nossa engenharia unificou esses dados sob regras de controle de acesso por perfil (RBAC), modelando um fluxo voltado a prevenir desastres geológicos e estruturais.

Atores e Escopo Mapeados

Morador: Registra ocorrências locais georreferenciadas e acompanha andamentos.
Agente: Executa vistorias estruturadas em checklists e dispara alertas à população.
Administrador: Modera acessos, audita logs e parametriza zonas críticas municipais.
Sis. Meteorológico: Fornece payloads via API para gatilhos automáticos de desastre.

Sequência: Registrar Ocorrência

O fluxo mais crítico do SIMAD — da interface do cidadão até a persistência no banco com protocolo imutável SIMAD-2026-XXXX.

Classificadores

Morador/Agente — inicia o fluxo
TelaRegistrarOcorrencia — exibe formulário
OcorrenciaController — valida e persiste
BancoDados — armazena com status 'Recebida'

Fluxo de Mensagens

1
Acesso à Interface: Morador inicia o registro enviando parâmetros via formulário ao OcorrenciaController.
2
Tratamento de Coordenadas: Se GPS falhar, a interface consome a API ViaCEP assincronamente para deduzir o georreferenciamento.
3
Geração de Payload: O controlador valida se os campos críticos e fotos estão no formato permitido.
Ver passos 4–6
4
Instanciação do Objeto: O controlador invoca o método new Ocorrencia() encapsulando os dados recolhidos.
5
Geração de Chave Única: O sistema calcula a chave de protocolo sequencial imutável SIMAD-2026-XXXX.
6
Resolução de Estado: O registro é persistido no MySQL com status inicial 'Recebida' e retorna confirmação de sucesso à tela.

Sequência: Fluxos de Alertas

O sistema foi projetado para ações humanas e para automações futuras — dois fluxos distintos, uma única arquitetura de despacho.

Alerta de Emissão Manual

Operador Autorizado

O agente define a mensagem de risco, a severidade (Informativo a Emergência) e os limites geográficos por bairro.

Fluxo Técnico

O AlertaController verifica a validade do token JWT de agente, persiste o alerta e aciona concorrentemente o serviço de mensagens.

Disparo Multicanal

É disparado um loop paralelo para despachar push notifications, SMS em lote e e-mails.

Alerta de Emissão Automática

Gatilhador Externo

Um payload JSON com os dados pluviométricos atuais é recebido do Sistema Meteorológico parceiro.

Filtro de Limiares

O sistema analisa e compara a milimetragem de chuva acumulada com os limites de segurança parametrizados.

Automação de Risco

Ultrapassado o teto técnico, o sistema instancia um alerta emergencial de forma nativa e inicia a notificação pública.

Estado Inicial

Toda ocorrência inicia como Registrada, gerado automaticamente pelo sistema (RN11).

Controle de Acesso

Somente Agente ou Administrador alteram o status. Moradores apenas consultam (RN13).

Rastreabilidade

Toda alteração de status gera um registro imutável em HistoricoOcorrencia com usuário e data/hora (CA03).

Persistência

Ocorrências não são apagadas fisicamente. Exclusão lógica preserva histórico para auditoria (RN14).

Tecnologias e Conclusão

Stack & Próximos Passos

Fase 1: MVP

Stack Desenvolvida: Frontend leve em JS Vanilla/CSS, Backend em PHP estruturado e MySQL nativo na hospedagem compartilhada da HostGator.

HTML5 · CSS3 · JS PHP Nativo MySQL HostGator

Justificativa Técnica: Rapidez na implantação, custo zero de infraestrutura e estabilidade ideal para validar a tese operacional.

Fase 2: Escalonamento

Evolução Arquitetural: Transição para Python 3.11 com FastAPI e persistência no Azure SQL Database.

Recursos Avançados: Firebase (push), Twilio (SMS) e roteamento inteligente via grafos.
